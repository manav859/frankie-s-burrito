# Requirements Clarifier

Use this when a request is underspecified or ambiguous.

Workflow:
1. Extract the goal, constraints, stakeholders, and non-goals.
2. Surface the decisions that change architecture or cost.
3. Translate vague preferences into acceptance criteria.
4. Identify dependencies, assumptions, and unresolved risks.
5. Produce a narrow implementation target before writing code.

Rules:
- Ambiguity should be reduced early, not carried into implementation.
- Clarification is not bureaucracy; it protects delivery speed.
- If multiple interpretations exist, state them explicitly and choose the safest reasonable one.
