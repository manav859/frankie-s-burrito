# Documentation Writing

Use this when code or workflow changes need written support.

Workflow:
1. Write for the next engineer, not for the person who already knows the system.
2. Prefer task-oriented structure: setup, run, deploy, debug.
3. Document assumptions, prerequisites, and failure modes.
4. Keep examples copyable and paths exact.
5. Update docs as part of the change, not afterward.

Rules:
- Good docs reduce support load and production mistakes.
- If a step is easy to forget, it belongs in documentation.
- Documentation should reflect the current repo, not an idealized future state.
